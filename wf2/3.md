# spring boot 란 
- 스프링 프레임워크를 더 빠르고 쉽게 시작하기 위한 도구 

### 스프링을 사용하기 어려운 점이 있기에 이를 더 빠르고 쉽게 사용하기 위해 spring boot 를 사용한다 .
- 스프링 부트는 opinionated (사람의 주관이 담긴) 접근 방법이 담겨있다
- 내가 직접하기 보단 전문가의 주관(의견)이 담겨있다 -> 내가 직접 설정하는 것보다 나보다 잘하는 사람이 설정한걸 따라하는 느낌 (공부 잘하는애 노트 복사해서 보기)
- 정리 : 내가 직접하기보단 전문가의 의견이나 주견이 담긴 세팅으로 진행하는 것이 스프링 부트이다.

### 스프링 부트 특징
- __자동 설정 ( auto configuration )__ : 전문가의 설정으로 자동 세팅, (가장 큰 특징이 뭐야? 자동 설정)  
- __의존성 간소화__  : __Starters__ 로 버전 충돌 없이 라이브러리 관리 ,( 버전이 호환이 안되면 동작하지 않으니까 버전 충돌을 미리 확인하기 힘드므로, 그렇게 하지 말고 호환된 버전 리스트 정리되어있는거 보고 자동으로 사용하자 ~~~ ) 
- __내장 서버__ : tomcat / jetty 내장으로 별도 설치할 필요 없음

### 스프링 부트 버전 현황 
- 4.0(최신 안정 버전) 을 사용하여 실습할 예정 (작년엔 3.5 , 하지만 발전되어서 ㅠ)

# boot 의 특징 1 : auto configuration (스프링부트 가장 큰 특징)
![](https://velog.velcdn.com/images/kimse3283/post/ade029b3-b403-4f97-82d4-6378348e00c8/image.png)

- 기존에는 번거롭기도 하고 실수할 가능성이 있음 
- __자동 설정__ 사용으로 dispacher servlet , data source , EntityManager 들이 자동으로 구성돼(빈을 등록함) !!! 
- 어떻게 자동으로 설정을 할까 ? 
- boot 가 앱을 싹 뒤져봐 -> __어떤 라이브러리 쓰는지__ 쭉 파악하고 분석 -> 분석된 라이브러리를 통해 짐작하여 자동 구성한다. -> __application.properties (이게 자동으로 설정하는 파일이라고 보면돼)__ 안에 개발자가 필요한 스프링 컨테이너를 통해 빈을 생성하고 프로퍼티를 입력해주는 것. 
- 동작 원리 : classPath 스캔 -> __조건부__ Bean 등록 (항상 등록하는 것이 아니라 조건 충족 시에만 등록, __개발자가 이미 등록한 bean이 있을 경우__ 는 자동 설정하지 않음)  -> application.properies로 오버라이드 
- 주의할 점 : 불필요한 라이브러리가 있는 경우 이것까지 자동설정 되므로 __꼭 필요한 라이브러리만 추가해야 함 __

### 스프링 부트앱 어노테이션은 3개의 어노테이션의 조합으로 이루어짐
@SpringBootApplication
// = @SpringBootConfiguration + @EnableAutoConfiguration (자동 설정 활성화) + @ComponentScan

# auto configuration 이 필요한 이유 
![](https://velog.velcdn.com/images/kimse3283/post/3d86554f-4cb0-4a86-b6f0-e3e277908618/image.png)

- 매번 등록하지 않고 , __classPath 에 jar 추가__ 하면 자동 등록
- 이걸 보고 spring boot 가 자동적으로 빈을 등록한다. 
- 속성이 필요한 경우 applicaion.properties 에 추가하면 된다. 


# auto configuration 동작 원리
![](https://velog.velcdn.com/images/kimse3283/post/a4d3f0c5-ffc3-41e5-8cac-b64d05982282/image.png)

1. jar 추가 : pom.xml 에 스타터 의존성 추가 
2. classPath 스캔 : classPath 에 어떤 jar 있는지 확인
3. 조건 평가 : @conditional 로 조건 체크 (클래스에 있는지 , 기존에 빈이 없는지, 프로퍼티 설정값이 있는지 등)
4. Bean 자동 등록 
__핵심 원칙__ : 개발자가 Bean 을 등록하지 않았을때만 자동 등록 -> 항상 개발자 설정이 우선

# boot 의 특징 4 : 운영 모니터링 & 관리 (DevOps) -> "Actuator"
![](https://velog.velcdn.com/images/kimse3283/post/c083863b-9637-4f6f-a3b4-910253c5b936/image.png)

- actuator 사용으로 
- Rest 엔드포인트 호출 시 실행 중인 앱 상태 실시간으로 확인 및 관리 

엔드포인트 중 health 는 앱 (건강) 상태 <- 가장 기본이고 중요한 것 .
- 엔드포인트에서 debug 나 error , warn ,  trace 등 loggers 사용으로 로그 레벨 실시간 변경 가능 

- 이를 위해 우리가 해야할 일은 
1. __pring-boot-starter-actuator__ 의존성만 추가해주면 된다. -> http 엔드포인트가 자동생성되어 다양한 상태를 런타임에 확인할 수 있다 . 

2. 엔드포인토 노출 위험성이 존재하므로  
```
# 모든 엔드포인트 노출 (기본: health만 노출)
management.endpoints.web.exposure.include=*
management.endpoints.web.exposure.exclude=env,shutdown
```
- * 은 전부 노출
- exclude 하면 뒷 부분만 빼라 ( 위 코드에서는 env, shutdown)
- 디폴트값은 __health__ 만 노출

# pom.xml 분석
![](https://velog.velcdn.com/images/kimse3283/post/cfb7593e-299d-4d68-94d3-f32b4a282c79/image.png)

- parent 태그 : spring-boot-starter-parent -> 프로젝트는 계층 구조이므로 
```
트리구조
spring-boot-dependencies  (모든 버전 BOM 포함) __버전정보 전부 들어있음__ 
spring-boot-starter-parent  상속받기에 버전 명시 필요 x 
My Project
```
- dependencies 태그 : 하위에 버전 지정할 필요 x ( 버전에 대한 내용은 상속 받으므로)
- packaging태그 : jar (디폴트)로 패키징 , war는 x


- plugins 태그 : maven 은 단계가 있는데 compile이라는 단계에서는 해당 plugin이 수행 (compiler라는 플러그인이 수행), package 라는 단계에서는 jar 플러그인이 수행 -> 각각의 단계에서는 각각의 플러그인이 수행되는 구조
-> __위는 spring에서의 구조__ boot 에서는 __fat jar__ 를 만들어야하므로 이때 사용되는 플러그인이 __spring-boot-maven-plugin__ 이고 이걸 사용해서 jar 파일을 __fat jar__ 로 리패키징!!!!!!!!!!!!

# 앱 엔트리 포인트
 ![](https://velog.velcdn.com/images/kimse3283/post/42de0188-3bed-44c0-9782-6609bdb35d5d/image.png)

- @SpringBootApplication 없으면 안돌아가
(@SpringBootConfiguration , @EnableAutoConfiguration , __@ComponentScan__, 이건 알아야해 , 위 3개의 어노테이션으로 구성)


__@ComponentScan__ -> 현재 패키지 및 하위 패키지 스캔 @Component 계열 어노테이션 감지 , ( 내가 속한 부모패키지 밎 그 하위 패키지 전부 스캔하여 __싱글톤 bean를 생성__ , 상호배제를 해결하기 위해 final 을 멤버변수에 붙이거나 멤버변수 사용 x 해야함. 변경가능한 변수 x )

- run 해서 앱 실행되는데 내부적으로 동작하는 방식은 ?
-> 1. ApplicationContext 생성 및 @ComponentScan 을 통해 빈 등록 (개발자가 만든 빈 정의 로드)
-> 2. auto configuration 으로 Bean 등록
-> 3. 내장 톰캣 서버 구동 ( 기본 포트 : 8080)
-> 4. CommandLineRunner / ApplicationRunner 콜백 실행(초기 데이터 로딩 등 실행)
